Firestore rules analysis for shared books prototype

Language/framework:
- Vite + React + TypeScript, Firebase Auth, Firestore Web SDK.

Firestore paths and queries:
- users/{uid}/transactions ordered by createdAt desc.
- books queried with where("memberIds", "array-contains", uid), sorted client-side.
- books/{bookId} updated directly to join by document ID and password.
- books/{bookId}/transactions ordered by createdAt desc.

Models:
- Transaction: type, amount, category, note, date, createdAt, createdByUid, createdByName, createdByPhotoURL.
- SharedBook: name, ownerUid, ownerName, joinPassword, memberIds, members, createdAt, updatedAt.
- BookMember: name, photoURL, role, joinedAt.

Access assumptions:
- Personal transactions are accessible only to their owning uid.
- Shared books are readable only by members.
- Shared book creation makes the creator the owner.
- Joining a shared book requires knowing the document ID and matching joinPassword.
- Members can read/create/delete transactions in a shared book.
- Password-based joining without a backend is a prototype design; a production version should move password verification to a trusted backend.

Attack review:
- Public list/read: denied by auth and membership checks.
- Unauthorized personal transaction access: denied unless uid matches auth uid.
- Unauthorized shared transaction access: denied unless parent book membership includes auth uid.
- Ownership hijacking on transaction create: denied by requiring createdByUid == auth uid.
- Schema pollution: denied with allowed-field validators.
- Large string DoS: string fields have max lengths.
- Join without password: denied because join update must preserve joinPassword equal to existing resource value.
- Join modifying other members: denied by allowing only the request uid member key to change.
- Query mismatch: app queries use only memberIds array-contains and createdAt orderBy, both aligned with rules.
